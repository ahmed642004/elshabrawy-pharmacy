# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Pharmacy e-commerce web app for **Elshabrawy Pharmacy** (client: Hend Elshabrawy). Online storefront for OTC cosmetics and supplements — no prescription products — with Cash on Delivery / card / mobile wallet checkout.

## Commands

```bash
npm run dev      # start dev server (Turbopack, http://localhost:3000)
npm run build    # production build
npm run start    # run the production build
npm run lint     # eslint
npx tsc --noEmit # type-check without emitting
```

No test runner is configured yet — there are no test files or a test script in `package.json`.

Environment variables live in `.env.local` (gitignored) and now point at a real Supabase project (`elshabrawy-pharmacy`, org `ahmed642004's Org`, region `eu-west-1`):

```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
```

`.env.local.example` documents the shape for anyone re-cloning. Without these set, `npm run build` fails while prerendering any page that creates a Supabase client (e.g. `/auth`, `/`, `/category/[slug]`, `/product/[slug]` — all now fetch from Supabase, see the **Backend** section below).

Schema and seed data live in `supabase/migrations/*.sql`, applied to the live project via the Supabase MCP's `apply_migration` (there is no local Supabase CLI/stack in this repo — migrations are hand-written SQL files that double as the applied history, not generated by `supabase db diff`). `lib/database.types.ts` is generated from the live schema (`generate_typescript_types`) and must be regenerated after any migration that changes the schema.

## Tech stack

- Next.js 16 (App Router, Turbopack), React 19, TypeScript, strict mode
- Tailwind CSS v4 — no `tailwind.config.js`; theme (colors, fonts) is defined directly in `app/globals.css` via `@theme inline`, following Tailwind v4's CSS-first config
- Supabase (`@supabase/supabase-js` + `@supabase/ssr`) for auth, database, and storage
- `lucide-react` for icons

### Next.js 16 gotcha

Dynamic route `params` (and `searchParams`) are async — always `await params` in a page/layout before reading its fields. See `app/product/[slug]/page.tsx` and `app/category/[slug]/page.tsx` for the pattern.

Supabase client rule: never instantiate `@supabase/supabase-js` directly in app code — always go through `lib/supabase/client.ts` (browser) or `lib/supabase/server.ts` (server), matching the `@supabase/ssr` cookie-based session pattern.

**`Header` is auth-aware** (fixed a real bug: the account icon used to always link to `/auth` even for a signed-in user, since `Header` had no session awareness at all). `Header.tsx` is now a thin `async` Server Component that calls `supabase.auth.getUser()` (`lib/supabase/server.ts`) and passes a `HeaderUser | null` down to `HeaderClient.tsx`, which holds all the original interactive markup/state (`menuOpen`, nav links, cart badge, mobile drawer) plus the account UI: logged out still shows the `<Link href="/auth">` icon unchanged, but logged in swaps it for an avatar button (first letter of `full_name`, falling back to email) that opens a small dropdown with the user's name/email and a **Sign out** button (calls the browser `supabase.auth.signOut()` then `router.push("/")` + `router.refresh()`). The mobile drawer's footer does the same swap (name/email + Sign out vs. the old "Sign in" button). Verified end-to-end with Playwright: logged-out → sign up → avatar shows correct initial → dropdown shows email, not a navigation to `/auth` → sign out → account icon reverts to linking `/auth` again.

**Search is real** — the desktop and mobile search inputs in `HeaderClient.tsx` used to be decorative (`<input type="search">` with no handler). Both are now wrapped in a `<form onSubmit={handleSearchSubmit}>` sharing one `searchQuery` state (harmless since only one input is visible at a time depending on viewport); submitting navigates to `/search?q=<encoded query>`. `app/(shop)/search/page.tsx` reads `q` (async `searchParams`, per the Next.js 16 gotcha above) and calls `searchProducts(query)` (`lib/queries.ts`), which does a case-insensitive `ilike` match across `name`/`brand`/`sub` (with `%`/`_` escaped so a literal percent sign in a query can't be misread as a wildcard) and returns `ListingProduct[]`, reusing the same shape/`ProductCard` grid and empty-state pattern as `CategoryListingClient`. An empty or missing `q` shows a "Search for a product" prompt rather than every product or an error.

**Live search dropdown**: `HeaderClient.tsx` also queries the browser Supabase client directly (`lib/supabase/client.ts`, not `lib/queries.ts` — that file is server-only, reads cookies via `next/headers`) as the user types, debounced 250ms via a `useEffect`+`setTimeout`, capped at 6 results. This is safe to run from the browser because `products` has a public-read RLS policy — no server round-trip/API route needed. A `searchQueryRef` guards against out-of-order responses (an in-flight request for an older query resolving after a newer one). `LiveSearchDropdown` is a component hoisted **outside** `HeaderClient` taking explicit props (`open`/`loading`/`results`/`query`/`onClose`/`onSeeAll`) — an earlier version defined it as a nested function inside `HeaderClient`'s render body, which the `react-hooks/static-components` lint rule correctly flagged (a new component identity every render resets state/remounts). Clicking a suggestion navigates straight to that product; the "See all results for…" row or hitting Enter goes to the full `/search` page. The one direct `setState` call in the debounce effect's early-return branch (clearing results when the input is emptied) has an inline `eslint-disable-next-line react-hooks/set-state-in-effect`, matching the same precedent already used in `lib/cart-context.tsx`'s hydration effect — this is a legitimate sync-to-input-value case, not a subscription to an external system.

**Route groups**: `app/(shop)/` holds every page that shares the standard site chrome (Home, Listing, Product Detail, Cart, Checkout) — its `layout.tsx` renders `Header`/`Footer` once instead of every page importing and rendering them itself (that was the original pattern; it got consolidated once all 6 pages existed and the duplication became obvious). `app/auth/` deliberately sits **outside** `(shop)` — it has its own minimal logo-only header and no footer at all, matching `Auth.dc.html`'s intentionally stripped-down chrome, so it can't share the `(shop)` layout. The `(shop)` folder name doesn't appear in URLs (Next.js route group syntax) — `/`, `/cart`, `/checkout`, etc. are unaffected. If a new page needs the standard chrome, add it under `app/(shop)/`; if it needs bespoke chrome like Auth, add it as a sibling of `(shop)` at the `app/` root instead.

Category id/label are no longer hardcoded — they come from the `categories` table via `getCategories()` (`lib/queries.ts`). `lib/categories.ts` now only holds what genuinely can't live in Postgres: `getCategoryVisual(id)` (a `Record<string, {icon, tone}>` map of Lucide icons + color tones, with a generic fallback for any category id not in the map) and `resolveCategorySlug(slug, validIds)` (maps old/alias spellings like `hair-care` → `hair` to a canonical id — this is pure URL-compatibility logic, not data, so it stays as code, but it's validated against the real category ids passed in from Supabase rather than a hardcoded list, so it can't drift out of sync with the actual catalog). `components/home/CategoryGrid.tsx` is now an `async` Server Component that fetches `getCategories()` directly and merges each row with its visual via `getCategoryVisual()`; `app/(shop)/category/[slug]/page.tsx` fetches categories once and passes them to both `resolveCategorySlug()` and `CategoryListingClient` (which takes `categories` as a prop now, not an import) so there's exactly one query per request, not two. `components/Header.tsx`'s `NAV_LINKS` still hardcodes its own slugs/labels (static nav config, not "category data") — the navbar's "Offers" link and "Vitamins & supplements" label are still an approximation of the underlying category model (no combined-category or deals-page concept exists yet); revisit once a real catalog/CMS defines categories properly.

`components/home/Hero.tsx` implements the design's default variant (`heroStyle: 'gradient'`, `accentLead: 'blue'`) — the source `.dc.html` also supports `duotone`/`minimal` hero styles and a teal-led accent via component props, which aren't exposed here since this is static marketing content, not a live-tunable component. The hero's "New in skincare" side panel and its copy are hardcoded from the design; swap it out once there's a real featured-collection data source.

`components/ProductCard.tsx` is marked `"use client"` because its add-to-cart button takes an `onClick` — any Server Component that renders it works fine (Next.js allows passing Client Components as children of Server Components), but the card itself can't be simplified back to a Server Component while that handler exists. It renders as a `<div>` with an absolutely-positioned stretched `<Link>` (not a wrapping `<a>`) specifically so the add-to-cart `<button>` can sit on top at a higher z-index — nesting a `<button>` inside an `<a>` is invalid HTML and was the original approach considered; don't revert to that. Its add-to-cart button calls `useCart().addItem(product)` directly (see the `lib/cart-context.tsx` note below) — there's no `onAdd` prop anymore. `components/home/ProductCarousel.tsx` now takes `products: Product[]` as a prop — `app/(shop)/page.tsx` (a Server Component) fetches it via `getPopularProducts()` (rows where `products.is_popular = true`); the carousel itself stays a Client Component purely for its scroll-arrow behavior.

`ProductCarousel` deviates from the source design here: the `.dc.html` only has a bare horizontally-scrollable strip (`overflow-x:auto`), but the client asked for explicit prev/next arrow buttons instead. It's a Client Component using a `scrollRef` + `scrollBy({ behavior: "smooth" })`, with the arrows disabled at each end (tracked via a `scroll` listener) and the native scrollbar hidden — touch/trackpad swipe still works via the underlying scroll container, the arrows are additive. If other horizontally-scrolling sections get built later (e.g. a similar carousel on Listing or Product Detail), match this arrow-button pattern rather than the design's bare-scroll one.

`components/Footer.tsx` was upgraded from a one-line copyright bar to the full footer in `Home.dc.html` (link columns + logo/copyright/legal bottom bar), since the design's "Home page footer" is really global site chrome, not Home-specific content — same reasoning as the navbar. Its link columns (`FOOTER_COLUMNS`) are rendered as plain text, not `<Link>`s, since most of those destinations (About us, Careers, Help center, etc.) don't have real pages yet; wire them up as those pages get built instead of linking to placeholders/404s.

**Listing filters are applied server-side**, not fetched-in-full-then-filtered-in-the-browser — this was a deliberate change from the original approach (which fetched all products via a since-removed `getAllProducts()` and filtered them client-side in a `useMemo`), made because that wouldn't scale once the catalog grows past a handful of products. `app/(shop)/category/[slug]/page.tsx` reads `cat`/`brand`/`price`/`sort` directly from its own `searchParams` (a second async param alongside the route's `params`, per the Next.js 16 gotcha above) and calls `getFilteredProducts({ categoryIds, brands, priceRangeIds, sort })` (`lib/queries.ts`) — every checkbox toggle round-trips to the database and gets back only the matching rows already sorted, via `.in()`/`.or()`/`.order()` on the Supabase query builder. `CategoryListingClient` no longer does any client-side filtering at all: `products` (the prop) **is** the filtered/sorted result, rendered as-is — no local `useMemo`, no `categoriesKey`/`brandsKey`/`priceRangesKey`.

Price-range buckets ("Under EGP 100", "EGP 300 – 600", etc.) are defined once in `lib/price-ranges.ts` (`PRICE_RANGES`, with `min`/`max`/`*Exclusive` bounds) and imported by **both** the client (for checkbox labels) and `lib/queries.ts`'s `buildPriceOrFilter()` (which turns selected range ids into a Postgrest `.or()` filter string, e.g. `and(price.gte.100,price.lte.300)`) — one definition so the label and the actual filter bound can't drift apart. The brand checkbox list comes from a dedicated `get_distinct_brands()` Postgres function (`0008_get_distinct_brands.sql`, called via `getBrands()`) rather than deriving distinct brands from whatever products happen to be in the current filtered result set — that would make the brand list itself shrink/shift as filters are applied, which is the wrong UX (facet options should stay stable; only the product grid should change). `SELECT DISTINCT brand` in Postgres also scales with the number of distinct brands, not the number of products, unlike the old client-side `Array.from(new Set(products.map(...)))` approach.

Filters (category/brand/price-range) and sort are still synced to the URL query string (`?cat=skincare,hair&brand=CeraVe&price=r1,r2&sort=price-asc`) via `useSearchParams`/`useRouter().replace` — there's no local `useState` for any of them, the URL is the single source of truth, so filtered/sorted views are bookmarkable and shareable, **and** it's what triggers the server refetch (changing the URL's query string causes Next.js to re-render the Server Component page with fresh `searchParams`, which is exactly the mechanism this relies on — no extra data-fetching wiring was needed beyond reading `searchParams` in the page). `catParam !== null` (present-but-possibly-empty) vs. `null` (absent) is the load-bearing distinction that lets "Clear all" show zero categories (`cat=` present-and-empty) without permanently losing the route's `initialCategoryId` default (only used when `cat` is absent from the URL entirely, i.e. on first arrival from a plain `/category/[slug]` link with no query string) — this logic is duplicated intentionally in both `page.tsx` (to build the actual filter query) and `CategoryListingClient` (to render correct checkbox state), reading the same URL. Because it calls `useSearchParams()`, the page wraps `CategoryListingClient` in a `<Suspense>` boundary, which Next.js requires for that hook. `filterSheetOpen` (the mobile bottom-sheet toggle) is the one piece of UI state that stays local — it's not filter data, so it doesn't belong in the URL.

The mobile "Filters" trigger and sort dropdown share one row; the source `.dc.html` has a CSS bug here (`justify-content:{{isMobile}}`, binding a layout property to a boolean), so the row layout (filter button + sort control) was adapted rather than copied literally — worth knowing if a future page conversion hits similarly-invalid inline styles in a `.dc.html` export, since they should be adapted, not reproduced.

`components/ProductCard.tsx`'s `brand` field (added for Listing) renders as a small uppercase line above the product name; Home's carousel products simply omit it, no visual change there.

`app/product/[slug]/page.tsx` now fetches the real product via `getProductBySlug(slug)` and calls Next's `notFound()` if it doesn't exist — every seeded slug (e.g. `vitamin-c-1000mg`, `blood-pressure-monitor`) renders its own real data now, and a bogus slug 404s instead of silently rendering the old CeraVe demo content. Only `cerave-moisturising-lotion` has rich seeded `description`/`dosage`/`ingredients`/`warnings`/`storage`/reviews (ported from the original static mock); other products have `null` content columns, and `ProductTabs`/`ProductReviews` fall back to generic default copy / "No reviews yet" when a field is null — see their `DEFAULT_CONTENT` map and empty-state branch respectively. `getRelatedProducts(categoryId, excludeSlug)` powers `RelatedProducts`, capped at 6 same-category products; the section is only rendered when there's at least one (`related.length > 0` in the page).

`ProductDetail.dc.html` has an artificial 700ms loading-skeleton delay (`setTimeout(() => this.setState({ loaded: true }), 700)`) purely for the design preview's feel — there's no real async fetch here to justify a loading state, so `app/product/[slug]/page.tsx` renders content immediately with no skeleton (the actual Supabase fetch happens server-side before the page streams, so there's nothing to skeleton over even now that it's a real query).

`RelatedProducts.tsx` follows `ProductCarousel`'s arrow-button pattern (per the note above) rather than the design's bare-scroll strip, but drops the design's implicit "view all" affordance — the source `.dc.html` has no "View all" button on this section (unlike Home's carousel), so the arrows are the only header-row control. It now takes `products: Product[]` as a prop instead of a hardcoded array.

The mobile-specific header in `ProductDetail.dc.html` (back-arrow + product name + cart icon, replacing the logo/hamburger) was **not** replicated — every page reuses the one shared `components/Header.tsx` instead, consistent with how this repo treats the navbar as global chrome rather than a per-page component. `ProductPurchasePanel`'s wishlist toggle is still local component state only (not persisted anywhere) — unlike add-to-cart, there's no shared "wishlist context." Its "Add to cart" button calls `useCart().addItem` with its qty stepper value and the route's real `slug` now (no longer a fixed demo slug). `rating`/`reviewCount` are typed `number | null` end-to-end (`ProductPurchasePanel`, `ProductReviews`) since only the seeded CeraVe product has a rating — components hide the star row or show "—" when null rather than assuming every product has reviews.

`ProductGallery` accepts an `images?: string[]` prop (from `product_images`, seeded empty for every product) and falls back to the original `Pill`-icon placeholder tiles when there are none — the client hasn't supplied product photography yet, so every product currently renders placeholders; wire in real URLs once images exist, no component changes needed.

### Cart (`lib/cart-context.tsx` + `components/cart/*`)

There is now a **real, shared cart** — not page-local demo data like everything else. `lib/cart-context.tsx` exports a `CartProvider` (wrapping the whole app in `app/layout.tsx`) and a `useCart()` hook with `items`, `savedItems`, `addItem`/`removeItem`/`updateQty`/`saveForLater`/`moveToCart`, and promo-code state. `Header`'s cart badge and every `ProductCard`/`ProductPurchasePanel` add-to-cart button all read/write this same context, so adding a product on Home, Listing, or Product Detail is reflected everywhere immediately (verified end-to-end with a Playwright script driving a real browser — add from 3 different pages, badge sync, qty/remove/undo/save-for-later/move-to-cart/promo, a full reload, then emptying the cart — all working, no console errors). `addItem`, `saveForLater`, and `moveToCart` all merge into an existing same-slug row (via a shared `mergeItemInto` helper) rather than pushing a duplicate — a self-review caught `saveForLater`/`moveToCart` originally missing this, which let the same product end up as two separate rows (duplicate React keys, double-counted totals) if it was saved for later and then re-added while still saved.

State is persisted to `localStorage` (`elshabrawy-pharmacy-cart` key) — written on every change, read back in a `useEffect` after mount (not during initial render) specifically to avoid an SSR/hydration mismatch, since the server always renders an empty cart. Derived totals (subtotal/delivery/discount/total) are **not** stored in the context; `OrderSummary` and `CartPageClient`'s mobile sticky bar each compute them from `items` + `promoApplied`.

**The cart is also server-synced for signed-in users** (guests stay pure-localStorage). A sync layer inside `CartProvider` listens to `supabase.auth.onAuthStateChange`: on sign-in (or page load with a session) it fetches the user's `cart_items` rows (migration `0013`; owner-only RLS; stores just `product_slug`/`qty`/`saved_for_later` and **re-joins `products` on load** so a synced cart always reflects current prices, unlike the localStorage snapshot), merges them with the local cart (union; on a slug conflict the local row wins — it's this device's most recent activity), then debounce-pushes (800ms) the full merged cart via the `replace_cart(p_items jsonb)` RPC (delete-all + reinsert in one transaction, security *invoker* so owner RLS is the enforcement; items whose product was deleted are skipped, not fatal) on every subsequent change. Pushes are held until the initial merge completes so a pre-merge local state can't clobber the server copy, and sync failures fail open (local cart keeps working). Two gotchas encoded in comments there: supabase-js builders are lazy thenables (a bare `void supabase.rpc(...)` never sends the request — `.then()` must be invoked), and `onAuthStateChange` callbacks are awaited internally by supabase-js, so Supabase calls inside them are deferred via `setTimeout` to avoid a deadlock. Verified end-to-end: add to cart → row lands in `cart_items`; wipe localStorage + reload → cart restores from the server at current prices.

Two deliberate simplifications vs. `Cart.dc.html`: no simulated ~450ms "updating" spinner on qty changes (same reasoning as skipping Product Detail's fake loading skeleton — no real async call exists to justify one), and the "No longer available" (out-of-stock) row styling in `CartItemRow` exists but has no live trigger in normal use — `ProductCard` already disables adding out-of-stock products, and nothing can flip an already-in-cart item to out-of-stock without real inventory data.

`CartPageClient` owns the undo toast itself (not part of the context) — on remove it captures the removed `CartItem` in local state before calling `context.removeItem`, auto-dismisses after 5s, and "Undo" re-adds it via `addItem` (this does not restore its original position in the list, only its presence/qty).

### Checkout (`app/checkout/page.tsx` + `components/checkout/*`)

**Checkout requires a signed-in account** — guest checkout was deliberately removed (it was originally supported; the client decided unauthorized users shouldn't be able to check out). `app/(shop)/checkout/page.tsx` calls `getAddresses()` (`lib/queries.ts`), which returns `{ addresses, isLoggedIn }`; if `!isLoggedIn` the page renders a "Sign in to check out" gate (icon, message, a `Link` to `/auth`) instead of `CheckoutClient` — a guest never reaches the step flow at all. This is enforced at the database layer too, not just the UI: `create_order()`'s `EXECUTE` privilege was revoked from `anon` (and from `PUBLIC`, since Postgres grants `EXECUTE` to `PUBLIC` by default on function creation — revoking just from `anon` wasn't sufficient, `anon` still inherited it through the `PUBLIC` grant) and the function itself now raises an exception if `auth.uid()` is null, so even a direct RPC call bypassing the UI is rejected. Only `authenticated` can call it now — see `0006_require_auth_for_checkout.sql` and `0007_revoke_create_order_from_public.sql`.

Steps are always **Delivery → Payment → Review** — the source `Checkout.dc.html` has a 4th "Prescription" step that only appears conditionally (`if cart has an rx item`), which is moot now that no products are ever rx. `CheckoutClient` owns all step state (selected/new address, payment method + card fields, step index) and validates the current step before advancing, the same inline-validation-on-submit pattern as `app/auth/page.tsx`. Payment step's promo code reuses `useCart()`'s existing `promoCode`/`promoApplied`/`applyPromo` rather than a separate implementation — if it was already applied on the Cart page, Checkout just shows it as applied.

`DeliveryStep`'s saved-address list is no longer a hardcoded `SAVED_ADDRESSES` array (there used to be a fake "Sara Ahmed" with two addresses shown to every user). Since `CheckoutClient` is now only ever rendered for a logged-in user (the page-level gate above), it takes just an `addresses` prop — no `isLoggedIn` prop needed anymore, that check lives entirely in `page.tsx`. It initializes `selectedAddressId`/`addingNew` from `addresses` instead of a fixed `"a1"`/`false` — zero saved addresses means it defaults straight to the "add new address" form.

A new address entered at checkout is **auto-saved** after a successful order — `placeOrder` calls `lib/actions.ts`'s `saveAddress()` (fire-and-forget, wrapped in `.catch(() => {})`) whenever `addingNew`, so a failure to save doesn't surface as an order error since the order itself already succeeded by that point. `saveAddress()` still checks `getUser()` server-side itself rather than trusting the caller, and marks the address `is_default` when it's the user's first one (checked via a `count`-only query before inserting).

Two guards in `CheckoutClient`: if the cart is empty (and no order was just placed), it shows an empty-state instead of the step flow; the **Back** button is always visible (not just on later steps) — on the first step it routes to `/cart` via the same `prevStep()` handler rather than decrementing past zero (this was caught by the Playwright verification below — an early version hid the Back button entirely on step 1, leaving no way back to the cart).

"Place order" now performs a **real insert**: `placeOrder` in `CheckoutClient` calls `lib/actions.ts`'s `createOrder()` server action, which RPCs Postgres's `create_order()` function (see **Backend** below) with the cart items + computed totals + payment method + a shipping snapshot (selected saved address or the "add new" form). The ~1s minimum "Placing order…" state is kept as a deliberate UX pause (`Promise.all([createOrder(...), timeout(1000)])`) even though the call itself is now real — it's no longer purely simulated, just floor-clamped so it never flashes instantly. The real order number returned by `create_order()` (format `EP-XXXXXX`, generated in SQL) replaces the old `Math.random()` one. On failure (network/RPC error) an inline error banner shows above the Place order button instead of silently succeeding; `orderPlaced` stays false so the user can retry. On success, `clearCart()` still runs and "Track your order" still routes to `/` (no order-tracking page exists yet — orders can be inspected directly in the `orders`/`order_items` tables for now).

Verified end-to-end with Playwright against the live Supabase project: empty-cart guard, full Delivery→Payment→Review flow (including card-fields-appear-on-select and validation-blocks-continue), promo code, place order → confirmation → cart cleared, and a real row landing in `orders`/`order_items` (confirmed via `execute_sql`). All checks passing, no console errors.

### Admin ops dashboard (`app/admin/` + `components/admin/*`)

Staff-only dashboard at `/admin` (Overview: KPI cards with real vs-yesterday deltas, recharts revenue bar chart, low-stock list, recent orders), `/admin/orders` (search/status-filter table + detail drawer that advances `placed→confirmed→delivered` or cancels), and `/admin/inventory` (search/category-filter table, atomic ±10 stock buttons, add/edit product modal **with image upload**). Key facts:

- **Access control is layered**: `proxy.ts`/`lib/supabase/middleware.ts` gates every `/admin*` request at the edge (logged-out → `/auth?redirect=/admin`, non-admin → `/`), and `app/admin/layout.tsx` re-checks via `getAdminSession()` as a backstop, rendering distinct logged-out vs. access-denied states. Admin = `profiles.is_admin` (migration `0009`), checked by a `security definer` `is_admin()` SQL function used in RLS policies (admins: SELECT-all + UPDATE on orders, SELECT-all on order_items/profiles, INSERT/UPDATE on products, INSERT/UPDATE/DELETE on product_images, and admin-only writes to the `product-images` storage bucket per `0012` — `0003`'s any-authenticated-user bucket writes were a security hole, since fixed). Mutations in `lib/actions.ts` (`updateOrderStatus`/`adjustProductStock`/`addProduct`/`updateProduct`) each re-check via a private `assertAdmin()` before writing and use `revalidatePath` (first use in this codebase) to refresh the server-rendered admin pages.
- **Numeric inventory**: `products.stock_count`/`low_stock_threshold`/`sku` (migration `0010`) back the dashboard; a `before insert/update` trigger derives the legacy `stock` enum (`out` ≤ 0 < `low` ≤ threshold < `in`) so every customer-facing read of the enum (`ProductCard` badges, filters) stays correct with zero component changes. Stock adjustment goes through an atomic `adjust_product_stock()` SQL function (`greatest(0, count + delta)` in one statement), not read-then-write, to avoid losing concurrent clicks — it's granted to `authenticated` but runs with *invoker* rights, so RLS makes a non-admin's direct RPC call match zero rows.
- **Product add/edit** share `components/admin/ProductFormModal.tsx` (edit mode is the same modal with a `product` prop). Add derives a unique slug from the name (`-2`/`-3` suffix on collision); edit deliberately never changes the slug (it's the public URL identity, present in carts and past order_items). Image uploads are re-validated server-side (`image/*`, ≤5MB, extension whitelisted to alphanumerics) in a shared `setProductThumbnail()` helper that writes the position-0 `product_images` row and best-effort deletes the replaced storage object.
- **Overview stats** (`lib/admin-stats.ts`) are pure functions over already-fetched data, mirroring `lib/cart-totals.ts`. All "today"/day-bucket math uses **Africa/Cairo** calendar days via `Intl` (`cairoDateKey`), and the 7-day chart walks days by UTC-anchored date arithmetic, *not* fixed 24h-millisecond subtraction — Egypt observes DST, so a milliseconds walk can skip/repeat a calendar day across a transition.
- Admin account: `is_admin = true` currently set for `ahmedabokhial2123@gmail.com` only (one-time data fix in `0009`). The header account dropdown shows an "Ops dashboard" link when the signed-in user is an admin (`Header.tsx` reads `profiles.is_admin` and passes `isAdmin` through `HeaderUser`).
- The auth page's `?redirect=` param is sanitized to same-origin paths (rejects `//`/absolute URLs) — it previously open-redirected. The design mock's "Prescription verified" badge was dropped (OTC-only domain rule), its `pending/processing` statuses map to the real `placed/confirmed`, and the sidebar's "pharmacist on duty" card is static/decorative.

## Backend (Supabase)

Project `elshabrawy-pharmacy` (`gjwkuhbhhueoxkmhoyrm`, `eu-west-1`, org `ahmed642004's Org`) is a real, live Supabase project — free tier. Schema + seed are hand-written SQL in `supabase/migrations/` (`0001_init_schema.sql`, `0002_seed_catalog.sql`), applied via the Supabase MCP; there's no `supabase` CLI/local stack in this repo, so any further schema change should be written as a new numbered migration file and applied the same way (`apply_migration`), then `generate_typescript_types` re-run to refresh `lib/database.types.ts`.

**Tables**: `categories`, `products` (+ `product_images`, `reviews`), `profiles`, `addresses`, `orders` + `order_items`. Enums: `stock_state`, `badge_tone`, `payment_method`, `order_status`. All tables have RLS enabled — catalog tables (`categories`/`products`/`product_images`/`reviews`) are public-read; `profiles`/`addresses` are owner-only (`auth.uid()`); `orders`/`order_items` are owner-read-only with **no direct client INSERT policy** — writes only happen through the `create_order()` SQL function below.

**`create_order(p_items, p_subtotal, p_delivery_fee, p_discount, p_total, p_payment_method, p_shipping)`** — a `security definer` Postgres function that inserts one `orders` row and its `order_items` in a single transaction and returns the generated order number. Requires a signed-in caller: raises an exception if `auth.uid()` is null, and `EXECUTE` is granted only to `authenticated` (revoked from both `anon` and `PUBLIC` — see the Checkout section above for why revoking from `anon` alone wasn't enough). `get_advisors(security)` still flags "authenticated can execute this security-definer function," which is expected and accepted here — that's the whole point, checkout requires being authenticated.

**`handle_new_user()`** trigger on `auth.users` insert — mirrors `raw_user_meta_data->>'full_name'`/`'phone'` (sent by `app/auth/page.tsx`'s `submitSignUp`) into a `profiles` row. `execute` on this function is explicitly revoked from `anon`/`authenticated`/`public` (a separate migration, `restrict_handle_new_user_execute`) since it should only ever run via the trigger, not be callable directly through PostgREST RPC — the advisor lint flagged this as unintentional and it was locked down.

The **canonical catalog** is the 16 products originally hardcoded in `CategoryListingClient` (seeded in `0002_seed_catalog.sql`), reconciling the three previously-separate hardcoded datasets described throughout this file. 6 are marked `is_popular = true` and back the Home carousel. Only `cerave-moisturising-lotion` has full detail-page content (description/dosage/ingredients/warnings/storage + 3 reviews, rating 4.6); every other product has null content columns and relies on the fallback copy described in the Product Detail section above. No product has seeded images (`product_images` is empty for all 16) — the client hasn't supplied photography yet.

**Storage**: a public `product-images` bucket (`0003_product_images_bucket.sql`) holds product photography. It's `public = true`, so reads go through Supabase's public-object URL endpoint and bypass RLS entirely — there is deliberately **no** `SELECT` policy on `storage.objects` for this bucket (`0004_drop_product_images_select_policy.sql` removed one after the security linter flagged it: a broad `SELECT` policy on a public bucket is both redundant for reads and additionally lets clients list/enumerate every file in the bucket via the API). Writes (`insert`/`update`/`delete`) are restricted to the `authenticated` role — there's no admin UI yet, so uploads happen via the Supabase dashboard or API directly.

**Images: `product_images` is the single source of truth** — `products.image_url` was dropped (`0005_drop_products_image_url.sql`) rather than keeping two separate places to point at the bucket. `position 0` in `product_images` doubles as the card/carousel thumbnail; every position (ordered) makes up the Product Detail gallery. `lib/queries.ts`'s `toProduct()` mapper derives `Product.imageUrl` this way, so every catalog query (`getPopularProducts`/`getAllProducts`/`getProductBySlug`/`getRelatedProducts`) joins `product_images(url, position)` and picks the lowest-position row — there is no per-product single-image column anywhere in the schema anymore. Once photos are uploaded to the bucket, just insert rows into `product_images` (position 0 = thumbnail); no component changes are needed, `ProductGallery`/`ProductCard` already render real `<img>` tags when a URL is present and fall back to the `Pill`-icon placeholder otherwise.

Both Supabase clients (`lib/supabase/client.ts`, `server.ts`) are typed with the generated `Database` type. `proxy.ts` (Next.js 16's renamed `middleware.ts` convention) + `lib/supabase/middleware.ts`'s `updateSession()` refresh the auth session cookie on every request so Server Components reading via `lib/supabase/server.ts` see a valid session.

Cart is a hybrid: `localStorage` remains the instant source of truth for everyone (and the only store for guests), with a server-sync layer on top for signed-in users via the `cart_items` table + `replace_cart()` RPC (migration `0013`) — see the Cart section above for the merge/push mechanics.

**Auth email confirmation is disabled** in the project's dashboard (Authentication → Sign In / Providers → Email → "Confirm email" off) — a deliberate choice, not an oversight. Supabase's built-in email sender (no custom SMTP configured) allows only a handful of confirmation emails per hour per project; testing the sign-up flow repeatedly hit `429 over_email_send_rate_limit`. With confirmation off, `supabase.auth.signUp()` returns an active session immediately — there is no OTP/email-verification step in the UI anymore (it was removed, not left half-wired). Before real customers sign up, this needs either custom SMTP (removes Supabase's rate limit) or re-enabling confirmation with that SMTP in place.

**No "welcome" screen** — `submitSignIn`/`submitSignUp` used to show a "Welcome back!"/"You're all set!" confirmation view before continuing; that view was removed entirely (no `"welcome"` in the `View` type, no `welcomeContext` state) per client feedback. Both now call `router.push(redirectTo)` + `router.refresh()` directly on success. `redirectTo` comes from a `?redirect=` query param (`useSearchParams`, so `app/auth/page.tsx` wraps the form in `<Suspense>` — same reason `CategoryListingClient`'s page does) and falls back to `/` when absent. This is what lets `app/(shop)/checkout/page.tsx`'s sign-in gate link to `/auth?redirect=/checkout` — a user who gets bounced to sign in while checking out lands back on `/checkout` after auth instead of on the homepage. Plain visits to `/auth` (e.g. from the header) have no redirect param and land on `/` as before. The `router.refresh()` after `push` matters here since `Header` is a Server Component reading the session — without it, a cached router segment could still show the logged-out header immediately after signing in (same reasoning as the sign-out flow in `HeaderClient`).

The auth page previously had a 6-digit OTP screen, but it was for **phone/SMS** verification, not email — Supabase has no built-in SMS sender at all (unlike the email one), it requires configuring an external paid provider (Twilio, MessageBird, etc.) before any phone OTP can be sent. Since there was no free/no-setup path to make it real, and the client hasn't asked for phone verification specifically, it was removed outright rather than left as UI-only dead code: no `otp` state, no `submitOtp`, no `"otp"` view. If phone verification is needed later, it needs a real SMS provider configured first, then `supabase.auth.signInWithOtp({ phone })` / `verifyOtp({ phone, token, type: "sms" })` wired in — not resurrected from git history, since the removal was thorough (mirrors how the earlier prescription-logic removal was handled).

## Internationalization (Arabic / RTL)

The **storefront is bilingual (Arabic + English) with Arabic as the default**; the **admin dashboard stays English/LTR** (internal staff tool). Built with **`next-intl` v4 in "without i18n routing" mode** — the locale lives in a `NEXT_LOCALE` cookie (default `"ar"`), **not** in the URL, so there are no `/ar`/`/en` route prefixes and none of the 13 routes had to move under `app/[locale]/`. `i18n/request.ts` (`getRequestConfig`) reads the cookie and loads `messages/{locale}.json`; `next.config.ts` is wrapped with `createNextIntlPlugin()`.

- **Messages** live in `messages/ar.json` + `messages/en.json`, namespaced per feature (`common`, `header`, `footer`, `home`, `productCard`, `listing`, `cart`, `checkout`, `product`, `search`, `auth`, `account`). Keep the two files at **full key parity** — the app throws on a missing key. Arabic plurals use ICU with all six categories (`zero/one/two/few/many/other`); English uses `one/other`. Use `useTranslations` in both RSC and Client Components, `getTranslations`/`getLocale` in async Server Components/pages, and `t.rich(...)` for embedded markup (each tag callback must return a **keyed** element, e.g. `(chunks) => <a key="terms">{chunks}</a>`, or React warns about missing list keys).
- **`<html lang dir>`** is set in `app/layout.tsx` from `getLocale()` (`dir="rtl"` for `ar`). `app/admin/layout.tsx` pins its shell (and both gate states) to `dir="ltr"` so an Arabic cookie never flips the dashboard, even though it renders inside the same root layout.
- **Language toggle**: `components/LocaleSwitcher.tsx` (mounted in `HeaderClient`'s desktop controls + mobile drawer) sets the `NEXT_LOCALE` cookie and calls `router.refresh()` — the same server-refresh pattern the header uses for sign-out. The label shows the *target* language.
- **Fonts**: `app/layout.tsx` loads **Cairo** (`next/font/google`, subsets `arabic`+`latin`, `--font-cairo`) alongside Public Sans/Inter. `app/globals.css` uses a **two-layer CSS-variable indirection** so the swap doesn't leak into the LTR-pinned admin: `:root` sets `--font-headline-src`/`--font-body-src` to the Latin fonts and `@theme` maps `--font-headline`/`--font-body` to those; `html[dir="rtl"]` overrides the two `*-src` vars to Cairo, and `html[dir="rtl"] [dir="ltr"]` (the admin subtree) restores them to Public Sans/Inter. Everything already uses the `font-headline`/`font-body` tokens, so no component-level font changes were needed.
- **RTL styling**: `dir="rtl"` auto-flips flex/grid; the manual work was converting physical Tailwind utilities to logical ones in the storefront (`ml-*`→`ms-*`, `mr-*`→`me-*`, `pl-*`/`pr-*`→`ps-*`/`pe-*`, `left-*`/`right-*`→`start-*`/`end-*`, `rounded-l/r-*`→`rounded-s/e-*`, `text-left/right`→`text-start/end`) plus `rtl:rotate-180` on directional icons (breadcrumb/link chevrons, carousel arrows, the Hero `ArrowRight`). Horizontally-scrolling carousels (`ProductCarousel`, `RelatedProducts`) also flip their scroll math under RTL (`scrollLeft` is negative in RTL — read via `Math.abs()` and flip direction using `getComputedStyle(el).direction === "rtl"`). Date formatting (`ProductReviews`, `OrderHistoryClient`) switches to the `ar-EG` `Intl` locale when the locale is `ar`.
- **Product data stays English**: product names, brands, and descriptions are **not** translated — Egyptian pharmacies keep medicine/brand names in Latin script (Panadol, CeraVe). Only UI chrome and **category labels** are translated.
- **Category labels come from the DB**: migration `0014_category_arabic_labels.sql` added `categories.label_ar` (backfilled for all 6 seeded categories). `getCategories()` returns `{ id, label, labelAr }`; storefront callers pick with `categoryLabel(category, locale)` (`lib/categories.ts` — Arabic when locale is `ar` and an Arabic label exists, else English). `CategoryGrid` (RSC, uses `getLocale()`), `CategoryListingClient`, and `FilterGroups` (Client, use `useLocale()`) all go through this helper; **admin inventory reads `categories(label)` directly (English)** and never calls it. Regenerate `lib/database.types.ts` after any category schema change (the `label_ar` column is already in the generated types).
- **Data-layer strings were de-Englished**: `getAddresses()` returns an `isDefault: boolean` (not a localized `"Default"`/`"Address"` string), `lib/price-ranges.ts` holds only ids/bounds (labels moved to `listing.priceRanges.*` keyed by range id), and `PAYMENT_METHODS`/`STEPS`/`SORT_OPTIONS`/`TRACKING_STEPS`/`SIDE_PANEL_ITEMS` keep only ids/keys with their labels in the message files. `lib/order-status.ts` **stays English** (shared with admin); the customer-facing `OrderHistoryClient` translates status labels via `account.tracking.*` keyed by the enum value.

## Domain rules

- **No prescription products** — the catalog is OTC cosmetics and supplements only. There is no prescription badge, filter, upload flow, or verification step anywhere in the app (deliberately removed — see the note above); don't reintroduce prescription UI unless the client's catalog scope actually changes.
- **Payment methods**: Cash on Delivery, card, mobile wallet — **UI only**, per the client, no payment processor is integrated (no Stripe/Paymob/etc. keys). `PaymentStep` collects card details but never sends them anywhere — only the chosen `payment_method` enum value is persisted on the order row, never card details.
- **Delivery zone/estimate logic**: not yet confirmed with the client — treat as open until specified. The checkout/cart flows currently use a flat placeholder rule (free delivery ≥ EGP 300, else a flat EGP 40 fee) in `lib/cart-totals.ts`, not a real zone-based calculation.
- **Checkout requires a signed-in account** — the client decided against guest checkout; unauthorized users are gated at `/checkout` (see the Checkout section above) and the database rejects `create_order()` calls with no session, so this is enforced at both the UI and DB layer. Browsing, cart, and adding to cart remain open to everyone — only placing an order requires auth.
